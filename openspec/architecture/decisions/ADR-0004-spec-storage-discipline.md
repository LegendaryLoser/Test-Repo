# ADR-0004 — Spec storage discipline (paper-derived)

## Context

A published result on transformer associative memory ("the paper") establishes
that semantic-recall systems exhibit:

- finite effective rank — distinct specifications collapse toward each other in
  embedding space;
- positive competitor mass — near-neighbor specifications are returned in place
  of the right one;
- power-law forgetting — older specifications are crowded out as the corpus
  grows;
- δ-convex lures — false recall cannot be eliminated by threshold tuning.

The repository is designed to *never depend* on semantic recall for spec
resolution. Specifications are addressed by stable identifiers and file paths.
The few places where semantic search is genuinely useful (human discovery)
require a continuous-measurement gate that turns the paper's negative result
into a CI signal.

## Decision

### 1. Identifier discipline

- Identifier format: `REQ-<DOMAIN>-<NNNN>`, where `DOMAIN` is uppercase
  alphanumeric ≤ 12 chars, `NNNN` is a zero-padded monotonic counter scoped to
  the domain.
- Identifiers are **immutable**. Renames are forbidden. Material change to a
  requirement produces a new REQ with `supersedes: <old-id>`; the old REQ
  remains in the file with `status: deprecated`.
- Identifier allocation is by `tools/spec_lint allocate <domain>` against
  `openspec/specs/INDEX.yaml`. Hand-allocation is permitted but checked.

### 2. Cross-reference discipline

- Every cross-reference uses **ID plus file path**, e.g.
  `REQ-SPEC-0001 @ openspec/specs/_meta/spec-storage.spec.md`.
- Illustrative prose references are a lintable defect — anything that names
  an artifact by description instead of by stable ID + path fails
  `prose-xref-banned`. Examples that fire the rule:

  ```text
  the auth spec
  the login requirement
  the original ADR
  ```
- The set of legal references is enumerable; broken references fail CI.

### 3. Atomicity

- One REQ contains one assertion. Compound requirements ("the system must do A
  and B") are split into `REQ-X-NNNN` and `REQ-X-NNNN+1`.
- Atomicity is enforced by spec-lint heuristic
  `compound-requirement-detector` (PHASE-1), plus by TEA's behavior
  decomposition: a REQ that decomposes into more than one Given/When/Then is a
  defect.

### 4. Anti-aliasing (the paper's δ-convex lures)

- Spec-lint computes pairwise n-gram similarity across all `Description` and
  `Acceptance` bodies in `openspec/specs/`. Pairs above a threshold are a
  lintable defect.
- The threshold and n-gram size are pinned in `tools/spec_lint/config.yaml`
  and reviewed quarterly (a `gate`-type task).

### 5. Storage format

Per-REQ block:

```markdown
## REQ-AUTH-0007
---
id: REQ-AUTH-0007
revision: 1
status: tests-green        # draft | tests-red | tests-green | reviewed | merged | deprecated
introduced: CHG-0042
supersedes: null
phase: PHASE-3
tier: integration          # unit | integration | e2e | stochastic
references:
  epic: EPIC-0005
  story: STORY-0019
  adrs: [ADR-0003]
---

### Description
<single assertion>

### Acceptance
- Given <state>, when <event>, then <observable outcome>

### Non-acceptance
- ...

### Notes
- ...
```

File layout: `openspec/specs/<domain>/<feature>.spec.md`. One file may contain
many REQ blocks for the same feature.

### 6. Index

`openspec/specs/INDEX.yaml` is auto-generated by `tools/spec_lint index`. It
is the **sole retrieval surface** for tooling. No tool we ship may use
embeddings or semantic search for spec lookup.

### 7. Append-only at the assertion level

A REQ that is `tests-green` and referenced by another artifact is frozen.
Material edits create a new REQ; cosmetic edits (typo, grammar) bump
`revision` and are allowed only if no downstream artifact refers to the prior
revision body.

### 8. Semantic search, if ever used

Semantic search is permitted **only** for human discovery UIs, never for
tooling correctness. If introduced, it must ship with a recall-under-
interference test (a synthetic corpus with controlled aliasing) whose
recall@k threshold is a CI gate. Drop in recall@k below threshold fails
the build.

## Consequences

- Renames cost more (must deprecate + add) — accepted, the alternative is
  catastrophic forgetting.
- Anti-aliasing lint will occasionally flag genuinely similar requirements
  that should be merged or distinguished — this is the desired outcome.
- A growing corpus does not degrade lookup correctness; deterministic
  path-based resolution is `O(1)` in interference.

## Compliance

- PHASE-1 delivers `tools/spec_lint` implementing rules 1–6.
- PHASE-2 wires the index regeneration into the pre-commit hook chain.
- `REQ-ARCH-0004` confirms the INDEX exists from PHASE-0.
