# CHG-0014 — `spec_lint index` subcommand + INDEX.yaml population

## Why

Closes audit finding **C1** (INDEX.yaml empty despite 14 live REQs).
[`ADR-0004` §6](../../architecture/decisions/ADR-0004-spec-storage-discipline.md)
names `openspec/specs/INDEX.yaml` the **"sole retrieval surface for
tooling"**. Today it is a `version: 0` scaffold with `requirements: []`,
while `openspec/specs/_meta/spec-storage.spec.md` defines 14 REQs
(REQ-SPEC-0001..0014). Anything that does ID lookup — PHASE-2 matrix
builder, future `req-coverage-100`, `phase-exit`, the `index-up-to-date`
gate itself — is starting from a broken substrate.

The PHASE-1 scope item for INDEX.yaml regeneration named the subcommand
`tools/spec-lint index` (note hyphen; CLI lands as `python -m
tools.spec_lint index`). It was never implemented.

## What changes

### `tools/spec_lint/index.py` (new)

Pure data module:

- `build_index_data(specs_root, repo_root) -> dict` — walks
  `specs_root` for `*.spec.md`, parses each, extracts every REQ block
  with valid frontmatter, returns the index data structure (deterministic
  YAML-serializable dict).
- `write_index(specs_root, repo_root, target_path) -> None` —
  serializes `build_index_data()` to YAML and writes to `target_path`.
- `check_index(specs_root, repo_root, target_path) -> int` — compares
  serialized expected against on-disk actual, ignoring the
  `generated_at` timestamp. Returns 0 if up-to-date, 1 if stale, 2 if
  the target file is missing or malformed.

### `tools/spec_lint/__main__.py` extended

New subcommand:
```
python -m tools.spec_lint index [--check]
```
- Default: regenerates `openspec/specs/INDEX.yaml`.
- `--check`: exits non-zero if INDEX is stale; does not modify files.

### `openspec/specs/INDEX.yaml` populated

Regenerated by running `python -m tools.spec_lint index`. After this
CHG lands, the file contains 14 entries (REQ-SPEC-0001..0014) with
the schema declared in INDEX.yaml's header comment:
```
version: 1
generated_at: <ISO8601>
requirements:
  - id: REQ-SPEC-NNNN
    file: openspec/specs/_meta/spec-storage.spec.md
    revision: 1
    status: tests-green
    tier: unit|integration
    phase: PHASE-1
    supersedes: null
    superseded_by: null
```

### `openspec/specs/_meta/spec-storage.spec.md` extended

- **REQ-SPEC-0015** — `python -m tools.spec_lint index --check`
  exits with status `0` against the current tree (INDEX.yaml matches
  what `build_index_data()` would produce, ignoring `generated_at`).

## Out of scope

- **CI wiring of `index --check`** as a gate — that is the
  `index-up-to-date` gate from [`ADR-0008` §1](../../architecture/decisions/ADR-0008-ci-gates-and-phase-exits.md),
  scheduled as a separate CHG (queued as CHG-0025 in the post-audit
  roadmap). When that CHG lands, `pip install + python -m
  tools.spec_lint index --check` becomes a step in `.github/workflows/
  ci.yml` between `validate` and `pytest`.
- **REQ-ARCH-0001..0008 indexing.** These REQs are defined in
  ARCHITECTURE.md §10 as bullet points, not in `*.spec.md` files
  (audit C2). The `index` subcommand walks `*.spec.md` only, so
  REQ-ARCH-* remain unindexed until CHG-0015 migrates them.
  INDEX.yaml therefore lists 14 entries (REQ-SPEC-0001..0014), not 22.

## Risk

- **YAML serialization determinism.** PyYAML's `safe_dump` sorts dict
  keys alphabetically by default; we use `sort_keys=False` and emit
  a fixed schema order so the file diff stays stable across
  regenerations. Tested explicitly.
- **`generated_at` exclusion from comparison.** Necessary so
  `--check` doesn't false-alarm on every commit. Trade-off: a
  human-edited `generated_at` (which would be a CLAUDE.md violation)
  is invisible to the gate. Acceptable; hand-editing INDEX.yaml after
  PHASE-1 is separately forbidden.
