# TASK-0050 — Methodology research note: dual metrics, coverage tautology, path-dependence, audit-type variance

## Intent

The user, on reviewing the headline ACGR series (Wave 5: ~91%, Wave 6: ~105%, Wave 7: ~87%) and the signal-ledger.md cluster-level analysis (42 dense clusters, with Wave 5+6+7 contributing 1 + 0 + 0 new clusters), proposed a dual-metric approach: coverage (breadth) + QD-score (depth), with adaptive depth allocation across cells sharing load-bearing observations.

Before adopting the dual-metric framework, the user surfaced three structural critiques:

1. **Coverage tautology.** If cells are defined post-hoc from observed clusters, coverage = 100% by construction.
2. **Ordering bias / path-dependence.** The cell map depends on which methods ran in which order; a different Wave-1 method choice would produce a different cell map and therefore different "load-bearing cell" identification.
3. **Audit-type criticality variance.** Axis D (severity bias) priorities are architecture-audit-specific; a UX audit or implementation audit would weight cells differently.

This TASK documents those critiques against the QD research literature, proposes a saturation-based reformulation that addresses them, and sets up the two-layer (domain-general / audit-specific) framework that the methodology codification ADR will need.

The output is a new staging document, `methodology-research-note.md`, that becomes input to the methodology codification ADR drafting. This TASK does NOT execute the cell-finalization work itself (originally planned as the TASK-0050 scope before the user's critiques re-shaped it); that work moves to TASK-0051 with the revised scope per `methodology-research-note.md` §5.

## Key conclusions

1. **The dual-metric approach is canonical in QD literature** (Pugh, Soros, Stanley 2016) and maps cleanly onto our framework. Adopting it is consistent with the field.
2. **The three critiques are well-founded and have research-literature names.** Coverage tautology = the "post-hoc behaviour-descriptor problem" (Cully 2018 §4.1). Ordering bias = MAP-Elites initial-population sensitivity (Mouret & Clune 2015 §5.1). Audit-type variance = domain-specific behaviour-descriptor (Stock 2025 §3.4).
3. **Mitigation: saturation in place of coverage.** Per-cell saturation operates within observed cells (no tautology, no |meaningful| denominator). Audit-level saturation = % of observed cells saturated. Combined with QD-score gives dual stopping: `saturation ≥ 0.95 AND ΔQD-score < 0.05 for two consecutive waves`.
4. **Mitigation: two-layer framework.** Domain-general layer (algorithm + metrics + admission rules) separated from audit-type-specific layer (axes + criticality assignment). Each audit type authors its own Layer 2.
5. **Mitigation: explicit path-dependence acknowledgment.** Document the method-sequence; treat the cell map as "the cell map that emerged from THIS sequence on THIS artifact"; don't claim universality. Restart-based mitigation (Mouret & Clune 2015) is more rigorous but ~N× expensive.

## Risk

- **The saturation-based reformulation has not been empirically validated against our corpus.** §3 proposes saturation as the replacement metric, but the actual per-cell saturation status hasn't been measured (that's TASK-0051's job). If saturation turns out to be tautologically trivial (all observed cells already saturated by Wave 4), the framework needs further reformulation. Unlikely but possible.
- **Layer 2 specification for architecture audit is not authored in this TASK.** The note describes what Layer 2 should contain (axes + criticality + cell admissibility list) but doesn't pin the architecture-audit instance. TASK-0051 will produce Layer 2's empirical pieces (per-cell saturation, QD-score); the formal Layer-2 spec belongs in the methodology codification ADR.
- **Citation quality.** The 12 references in §7 are accurate per my training-data recall; one is flagged unreliable (arXiv 2602.09341). The methodology codification ADR's drafting should re-verify all 12 against current bibliographic databases before publication.
- **The Layer 1 / Layer 2 separation introduces specification overhead.** Each audit type now requires a Layer-2 specification before it can run. This is methodologically clean but slows audit initiation. Pragmatic mitigation: Layer 2 specs can be inherited from similar audit types and incrementally refined.
