## Findings deliverable (final assistant message)

I have enough material. Now I'll write the retrospective.

INTRRETRO-CRIT-001 — The mandatory four-line commit trailer block (`Task`, `Requirements`, `Tests-Status`, `Phase`) cited in ADR-0005 §3 was, by 2029, the single most-resented ceremony on the team. Looking back from 2029, this cost the team roughly 30-45 seconds of cognitive overhead per commit times ~9,000 non-checkpoint commits across three years — call it 80-100 engineer-days of pure typing — plus an uncounted but larger cost in rebases-to-fix-rejected-trailers (the `commit-trailers-valid` pre-receive hook is unforgiving). The benefit was real: deterministic matrix rebuilds and an audit trail that survived two compliance reviews. But by mid-2027 every senior dev had a git template or a shell alias that filled all four fields automatically, and the `Tests-Status` field in particular degenerated into "green" by default because nobody actually thought about the red→green transition at commit time — they thought about it once, in the squash. The ratio was **not worth it as specified**. If we could re-architect from 2026, we would have made trailers OPTIONAL on feature-branch commits and MANDATORY only on the squash-merge commit to main, with the matrix builder reading the squash trailer plus the PR description rather than every WIP commit.

INTRRETRO-CRIT-002 — The `red-before-green` gate (ADR-0005 §4, ADR-0006 §4, ADR-0008) was enforced literally — `tools/trace/validate_commit.py` walks history to confirm a prior commit had the test in a failing state. In practice the team learned to write a red test, commit it with a fake assertion that fails (`assert False, "red"`), then immediately commit the real implementation. The "red" commit was theatre roughly 70% of the time, per a Q3-2028 informal sampling. The gate prevented exactly four real "test was always green" regressions in three years, and cost us roughly 2 engineer-weeks per quarter in commit-shuffling, lost-stash incidents, and "I forgot to commit the red first" rebase fights. Ratio: **not worth it**. We'd replace it with a PR-level check that a test file was first introduced in a CI run where it was failing — same guarantee, no commit-graph contortion.

INTRRETRO-SER-003 — The "no mocks of in-repo modules" rule (ADR-0006 §2) was the single best decision the architecture made for code quality and the single worst decision for unit-test runtime. Tests that previously ran in 8 seconds now run in 95 seconds because the "real adapter" for Sheets requires REST round-trips. The team's CI bill tripled between 2027 and 2028. The benefit was equally real: zero test-double-drift incidents, and three production bugs that were caught by integration tier that would have been masked by mocks. Ratio: **worth it but with significant amendments** — we'd permit a narrow class of in-process real-fakes (e.g., an in-memory Sheets adapter that is itself production code with its own test suite), reclassified from "mock" to "alternate real adapter." ADR-0006 §3 hinted at this and we should have leaned into it harder.

INTRRETRO-SER-004 — The `anti-aliasing` lint rule (ADR-0004 §4) — pairwise n-gram similarity across all REQ descriptions — produced an estimated 1 true positive for every 8 false positives across 2027-2028. The team developed a Pavlovian "just rephrase until the linter shuts up" reflex, which often made the requirement *less* clear in the name of being less similar to its neighbor. Cost: ~3 engineer-days/quarter triaging false positives and rewording perfectly clear requirements. Benefit: caught the genuine REQ-AUTH-0042/REQ-AUTH-0051 duplication that would have produced an ambiguous test contract. Ratio: **worth it but mis-tuned**. We'd raise the threshold by 30%, scope it to within-domain rather than corpus-wide, and add an explicit "yes I know these are similar, here's why" exemption file.

INTRRETRO-SER-005 — ADR-0007's "no recorded fixtures" rule for stochastic tests was right on principle and operationally painful. Every PR that touched LLM-adjacent code had a 5-15 minute test wait for real API calls and a non-trivial chance of a flaky failure on a distributional assertion that was passing 19 times out of 20. The team eventually invented an unofficial `@flaky-rerun-once` decorator that the architecture forbade in spirit but that nobody had the energy to litigate. Cost: ~$18k/year in API spend on tests, plus ~6 engineer-weeks/year of PR-cycle-time waste. Benefit: caught two model-regression bugs (one on Opus 4.6→4.7) that fixture-replay would have masked. Ratio: **worth it with amendments** — we should have formally sanctioned a `tests/stochastic-cached/` tier that replays for unchanged prompts and only re-runs live when the prompt or model pin changes.

INTRRETRO-CRIT-006 — Checkpoint commits from the `Stop`/`SessionEnd` hook (ADR-0005 §7) polluted git history catastrophically. After three years and many sessions per day, `git log --oneline` on long-running branches was 60-80% checkpoint commits. The team developed branch-local squash conventions that defeated the audit-trail purpose of the checkpoints in the first place. Cost: every `git bisect` was a minor nightmare; new hires routinely lost work trying to "clean up" the checkpoints before push. Benefit: prevented two actual container-reclaim losses in three years, both in PHASE-2 before the team had internalized the manual-commit habit. Ratio: **not worth it as designed**. We'd move checkpoints to a separate ref namespace (`refs/checkpoints/<session>`) invisible to `git log` and gc'd after 30 days.

INTRRETRO-SER-007 — ADR-0004 §1's identifier immutability rule ("Renames are forbidden; supersession is the only path") was correct in principle and produced exactly the named-thing-vs-meant-thing drift it was designed to prevent. In practice, when we renamed a domain (e.g., `BILLING` → `INVOICING` in Q2 2028) the supersession dance produced ~140 new REQ-IDs with `supersedes:` pointers and required updating every Epic, Story, and test annotation. It took a 3-engineer sprint. Cost: ~12 engineer-weeks across three rename events. Benefit: every historical REQ-ID remained resolvable; no audit ever broke. Ratio: **worth it overall** but we'd add a sanctioned bulk-rename CHG template with tooling, instead of forcing each rename through one-at-a-time supersession.

INTRRETRO-PROC-008 — The `prose-xref-banned` lint (ADR-0004 §2) — flagging phrases like "the auth spec" — fired on PR descriptions, commit messages, *and* docstrings, including in places where the prose reference was genuinely the most readable form. Cost: low, maybe 1 day/quarter. Benefit: low; the rule caught real ambiguity perhaps 3 times in three years. Ratio: **keep the constraint but scope it** — only enforce inside `openspec/` files, not in code comments, commit bodies, or PR descriptions. Most of the team had an unofficial "type the ID first, then the prose, separated by `@`" muscle memory that satisfied the rule without thinking.

INTRRETRO-CRIT-009 — The CHG envelope structure (one PR per CHG with REQs scoped inside) actively hindered cross-team coordination. When the auth team, billing team, and Apps Script team needed to land a coordinated change in Q4 2027, we burned roughly 4 weeks figuring out whether it was 1 CHG-with-3-task-streams or 3 CHGs-with-cross-references. ADR-0004 and ADR-0005 don't address multi-team CHGs; the silence meant every multi-team change re-litigated the question. Cost: ~6 engineer-weeks across three years on coordination overhead that should have been pre-decided. Ratio: **not worth it as silent**. We'd add an `EPIC-as-coordination-envelope` pattern explicitly, with sub-CHGs allowed under one Epic, and a documented multi-team review path.

INTRRETRO-SER-010 — ADR-0008's `gate-coverage` meta-gate ("every principle has a gate, every ADR has a gate") was philosophically beautiful and produced one concrete failure mode: every time someone wrote a new ADR, they had to invent a gate even when no real check was warranted. We have approximately 4 "ceremonial" gates that pass unconditionally because their owning ADR was an architectural statement, not a verifiable invariant. Cost: low individually, but compounded into a "well, just add a TODO gate" cargo-cult by 2028. Ratio: **keep but refine** — explicitly permit `enforces: documentary-only` for ADRs that are guidance rather than invariant, with `gate-coverage` accepting that classification.

INTRRETRO-PROC-011 — The push invariant amendment to ADR-0005 §6 (per-commit push, per-merge sync) was a direct response to a real incident on `claude/general-session-KXgas`. It worked. It also made every commit ~600ms slower (network round-trip) and broke laptop offline work for two team members entirely. Cost: real friction, ~0.5% of dev time. Benefit: zero post-amendment commit-loss incidents. Ratio: **worth it** but we'd add an offline-mode escape valve (push-on-reconnect) and the amendment should have shipped with that on day one.

INTRRETRO-SER-012 — The vendored BMAD install (ADR-0002 §4, 2.6 MB committed) was the right call for reproducibility and the wrong call for code review. Every BMAD upgrade PR (we had 5 in three years) was a several-thousand-line diff that no human meaningfully reviewed. The "smoke test" safeguard (ADR-0002 §8) caught one real regression in 2028 (a removed skill); the manifest-diff gate caught it too. Cost: ~1 engineer-day per upgrade PR pretending to review vendored markdown. Benefit: actual reproducibility and zero session-start network dependency, which paid off during a Q1 2028 npm registry outage. Ratio: **worth it** but we'd auto-collapse the vendored diff in the review UI and require human review only on the manifest and config.

INTRRETRO-CRIT-013 — The `req-coverage-100` gate (ADR-0006 §5, ADR-0008) — every non-deprecated REQ must have ≥1 passing test — was the constraint most-routed-around in the codebase. When a feature genuinely didn't need a test for a given REQ (a documentation REQ, a procedural REQ), the team would either (a) deprecate the REQ prematurely or (b) write a tautological test that asserted `True`. We counted 23 such tests in a Q1 2029 audit. The rule was sound for behavioral REQs; it was theatre for non-behavioral ones. Cost: ~4 engineer-weeks/year of test-shaped fiction. Benefit: caught real coverage gaps in PHASE-3 and PHASE-5 stand-up. Ratio: **not worth it as a 100% gate**. We'd introduce a `tier: documentary` classification exempt from `req-coverage-100` but tracked separately.

INTRRETRO-SER-014 — Onboarding time for a new contributor was 3-4 weeks to first independent PR in 2027, dropping to 2 weeks by 2029 once we wrote (unofficially) a "what you actually need to know" doc that distilled the 8 ADRs into a 6-page primer. The load-bearing knowledge: the trailer block, the CHG envelope, the four test tiers, and the no-mock rule. Everything else was lookup-on-demand. Cost: ~1 engineer-week of senior time per new hire just to walk them through the discipline. Cargo-cult items every senior failed to articulate cleanly when asked: (a) *why* the anti-aliasing threshold is what it is — nobody on the team remembered, the "quarterly review" gate (ADR-0004 §4) had not actually run since 2026; (b) *why* `Tests-Status: red→green` requires the arrow rather than separate commits — nobody could explain it. Ratio: **architecture made the codebase legible but the discipline opaque**. We'd commission a maintained "Operating the discipline" cheatsheet as a first-class deliverable, not a wiki page.

INTRRETRO-SER-015 — Hiring impact was net-positive but smaller than the architecture-tax would suggest. Candidates who toured the codebase commented favorably on the legibility (the ID-everything discipline really does pay off in code-reading). The discipline scared off 2-3 candidates out of ~40 interviewed across three years (per our recruiting notes), all of whom were senior ICs uncomfortable with the ceremony. Net: probably a wash on hiring, with a slight selection toward people who actually like rigor — which is what we want. Ratio: **worth it**, no amendment needed.

INTRRETRO-PROC-016 — The "single architect signs ADR amendments" bottleneck (implicit in CLAUDE.md's "ADR amendment or deferred task" rule) became a real queue in 2028. We had ~7 amendments waiting on Winston's review at the peak, blocking three teams. Cost: ~3 engineer-weeks of effective blocking. Benefit: ADR consistency was high; we never had two contradictory amendments land. Ratio: **keep the constraint but scale it** — designate two backup ADR-reviewers, with a publicly tracked queue and SLA.

INTRRETRO-CRIT-017 — The `Stop`/`SessionEnd` checkpoint hook firing a network push (ADR-0005 §6 amendment + §7) on every session end created a cascading dependency: a flaky network meant a session couldn't end cleanly, which meant the journal `session_end` event never appended, which meant the next session's `SessionStart` resume hook treated the prior session as still-active and presented a confusing UI. Estimated lost engineer-hours from this specific failure mode: ~2 weeks/year. Benefit: covered the committed-but-unpushed reclaim case. Ratio: **not worth it**. The push and the session_end append should be independent operations; the journal close should succeed even if the push doesn't, with push retried in the next session's SessionStart.

INTRRETRO-PROC-018 — Capacity-to-ship: by our count, **roughly 5 features** were materially delayed (≥2 weeks beyond honest estimate) by architecture-discipline overhead that, in hindsight, didn't pay off. They were: (1) the Q3 2027 billing integration delayed by the multi-team CHG question (see INTRRETRO-CRIT-009); (2) the Q1 2028 Sheets adapter rewrite delayed by anti-aliasing false positives; (3) the Q2 2028 model upgrade delayed by stochastic-tier flakiness; (4) the Q4 2028 reporting feature delayed by the `red-before-green` rebase loop; (5) the Q1 2029 multi-project shared-types refactor blocked on ADR-amendment queue. Out of ~60 features shipped in the period, 5 meaningfully delayed is an 8% drag — measurable but not catastrophic.
