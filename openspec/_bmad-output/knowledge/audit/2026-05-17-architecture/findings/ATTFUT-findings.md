## Findings deliverable (final assistant message)

I have full coverage of the architecture documents. Now I'll produce the forward-looking attitudinal review.

ATTFUT-CRIT-001 — **Credentials / service-account lifecycle ADR is missing.** ADR-0003 §3 requires integration and E2E tiers to hit "real Sheets/Drive REST adapters against dedicated test artifacts owned by a CI service account" and PHASE-5 names a "CI service account" owning a `dev` GAS deployment. ADR-0007 mandates real Anthropic API calls. ADR-0006 §2 permits substituting "HTTP transport at the network boundary." Nowhere is it specified: who holds the service-account key, how it is rotated, how a leaked token is revoked, how a service account is bound to a project, where the canonical reference of "this account owns these artifacts" lives. **Addition required:** ADR-0009 "Credentials and service-account lifecycle" plus REQ-ARCH-0009 ("every external system access uses a named service account with a rotation policy and a revocation runbook") plus a gate `credentials-inventory-current`. **Consequence of not adding:** by PHASE-3 the architecture has live Anthropic keys, by PHASE-5 it has GAS service-account keys, and no SoT governs them — a leak becomes an undefined response, which violates P1.

ATTFUT-CRIT-002 — **No ADR codifies the meta-methodology that produced ADRs 0001-0008.** ARCHITECTURE.md §12 states "if it isn't an ADR, it isn't decided", and §1 demands one CI gate per principle. The methodology by which an ADR is authored, reviewed, accepted, superseded, or amended is itself a decision — and it is undocumented. ADR-0002 already shows two amendment-log entries and ADR-0005 shows one, with no specification of who may add an amendment row, what "material change" means, or how an amendment is reviewed versus a new ADR. **Addition required:** ADR-0009 (or 0010 if 0009 is the credentials ADR) "ADR lifecycle and meta-methodology" with REQ-ARCH-0010 ("every ADR carries status, supersession links, and amendment-log discipline; amendments are append-only and atomic"), plus a gate `adr-lifecycle-valid`. **Consequence:** the amendment mechanism rots into a freeform notes field within two phases.

ATTFUT-CRIT-003 — **Phase-0 re-gate procedure is named but unspecified.** ARCHITECTURE.md §1 states "Modifications to the structure described here require an ADR or a `Material-Architecture-Change: true` commit trailer and a Phase-0 re-gate." Neither `Material-Architecture-Change` nor "Phase-0 re-gate" is defined anywhere. ADR-0008's `phase-exit` gate only checks REQ-ARCH-* `tests-green` on merge to main — it has no notion of revisiting an already-passed phase. **Addition required:** procedure `PROC-PHASE0-REGATE` (canonical home `openspec/architecture/procedures/`, which itself does not exist as a directory in §3) defining the trailer's semantics, the diff scope it forces re-evaluation over, and the gate `material-arch-change-detected` that fires when the trailer appears. **Consequence:** the "frozen architecture artifact" can be edited silently because the enforcement mechanism it cites does not exist.

ATTFUT-CRIT-004 — **Cross-document "verbatim" consistency lint is absent.** REQ-ARCH-0007 requires "Root `CLAUDE.md` cites the five principles verbatim." CLAUDE.md (root) already restates the five principles, ARCHITECTURE.md §1 carries the canonical list, and there is no gate that compares them character-by-character. Worse, ADR-0008 declares ARCHITECTURE.md §7 a summary of ADR-0005, ARCHITECTURE.md §8 a summary of ADR-0005 §8, and ARCHITECTURE.md §9 a summary of ADR-0002 — all "verbatim/summary" surfaces with no equality check. **Addition required:** gate `verbatim-block-equality` plus a `<!-- verbatim:source=... -->` marker convention and a `summary-block-source-current` lint. **Consequence:** CLAUDE.md and ARCHITECTURE.md will drift on the principles within the first edit cycle — a direct P1 violation in the very document that defines P1.

ATTFUT-CRIT-005 — **Gate severity classification is unspecified.** ADR-0008 §1 inventories ~26 gates with `Trigger` (every PR / nightly / pre-commit) but no severity (blocking vs warn vs informational). ADR-0008 §6 names `Checkpoint: true` as a "partial-bypass" against "a specific subset of gates listed in `tools/trace/checkpoint_exemptions.yaml`" — that file is referenced but never specified. Other implicit bypasses (e.g. `Bootstrap:` trailer in ADR-0006 §4) carry no severity model. **Addition required:** gate metadata extension and a `gate-severity-consistent` meta-gate; REQ-CI-* requirements with a `severity:` field; specification of `checkpoint_exemptions.yaml` as a tracked artifact under spec_lint coverage. **Consequence:** by PHASE-2 the gate inventory has 30+ entries with ad-hoc bypasses; without a severity taxonomy, "is this gate blocking?" becomes folklore.

ATTFUT-SER-006 — **Promotion procedure for `_bmad-output/` drafts is named but not specified.** ADR-0002 §7 defines promotion as "the act of moving a draft from staging to its canonical home" and says it is "a `docs`-type TASK under an explicit CHG." No procedure file defines task structure, no gate verifies promotion completeness, and the deferred `stale-staging` rule has no SLA threshold. **Addition required:** `PROC-DRAFT-PROMOTION` procedure and gate `promotion-task-complete` (verifies the canonical artifact exists, has frontmatter, and has matrix entry on TASK close). **Consequence:** drafts accumulate; the "non-authoritative" namespace becomes a parallel SoT — exactly the violation §7 was written to prevent.

ATTFUT-SER-007 — **Audit corpus persistence is unenforced.** ADR-0005 §5 defines an append-only JSONL journal; ADR-0005 §9 defines `tools/trace/audit.py`. ARCHITECTURE.md §3 places journals in `.claude/journal/` and the .gitignore in PHASE-0 §11 lists `.claude/journal/` as gitignored runtime. The journal is the deepest source of truth (ADR-0005 §1 ordering), yet it is *never persisted off the container*. Container reclaim deletes it. **Addition required:** REQ-AUDIT-0002 "session journal is uploaded to durable storage at every checkpoint commit," plus ADR-0011 "Audit corpus durability" naming the storage substrate (artifact upload, blob store, or commit-as-notes-ref), plus gate `journal-uploaded-on-checkpoint`. **Consequence:** ADR-0005's stated worst-case ("zero loss on container reclaim if push completed") understates reality — the journal layer is lost regardless of push state, breaking the four-source-of-truth model from the bottom up.

ATTFUT-SER-008 — **No observability ADR; zero metrics or dashboards spec'd.** Every phase exit, gate, and amendment is reportedly enforceable but nothing makes the *state* visible. ADR-0007 §5 mentions "cache hit rate is reported per CI run" and ADR-0008 mentions `cost-budget` aggregation, yet no dashboard, alert, or retention policy is defined. **Addition required:** ADR-0012 "Observability and operational metrics" enumerating mandatory series (gate pass-rate per ADR, cost burn-down per stochastic test, journal-write rate, matrix-drift fire-count, checkpoint-exemption fire-count), plus REQ-OBS-0001 ("each gate emits a structured metric record on every fire"). **Consequence:** by PHASE-3 the team is debugging cost overruns blind; by PHASE-5 there is no signal that any gate is silently misfiring.

ATTFUT-SER-009 — **Cost-budget kill-switch is missing.** ADR-0007 §4 says "exceeding the budget fails the build" but defines no per-day, per-month, or per-account hard cap, and no mechanism to halt stochastic tests when an account is bleeding. The architecture is one runaway test loop away from a five-figure bill. **Addition required:** ADR-0013 "Cost containment and kill-switch" with REQ-LLM-* requirement "anthropic-client refuses calls when day-budget exceeded," plus gate `kill-switch-armed`. **Consequence:** PHASE-3 ships a system whose only cost control is post-hoc CI failure — a CI gate cannot prevent spend that already occurred.

ATTFUT-SER-010 — **CHG and TASK lifecycles are entirely undocumented.** ARCHITECTURE.md §4 names `CHG-<NNNN>` and `TASK-<NNNN>` as atomic units but never defines their state machine. ADR-0002 mentions `docs`-type TASKs, ADR-0004 mentions `gate`-type tasks, PHASE-3 mentions a `gate`-type task for prod promotion — task *types* exist as folklore but are nowhere enumerated. CHG status (open / in-flight / merged / abandoned) is similarly undefined. **Addition required:** ADR-0014 "CHG and TASK lifecycle"; REQ-LIFE-0001/0002 enumerating states and legal transitions; gate `chg-task-status-valid`. **Consequence:** by PHASE-2 the traceability matrix has TASKs in undefined states that the audit tool cannot reason about.

ATTFUT-SER-011 — **Parallel-convention detector is implied by P5 but not implemented.** Principle 5 says "no parallel convention is introduced," and the same admonition reappears in CLAUDE.md. There is no gate that mechanically detects when, say, `docs/`, `notes/`, or `_drafts/` appears, or when a new `*.spec.yaml` alternative format is introduced. ADR-0008's `top-level-allowlist` covers only the root, not nested aliasing. **Addition required:** gate `parallel-convention-detector` (a generalisation of `top-level-allowlist` that scans for known-aliasing patterns: e.g. any directory matching `(docs|notes|drafts|plans|designs)/` outside the sanctioned homes). **Consequence:** P5 is unguarded; the first sub-team that needs scratch space invents `notes/` and the SoT discipline begins eroding directory-by-directory.

ATTFUT-SER-012 — **Incident-response and rollback procedure is undefined beyond BMAD upgrades.** ADR-0002 §8.4 names a BMAD-specific rollback ("`git revert <upgrade-sha>`"). No general incident-response ADR covers: a green test suite that hides a production regression, a CI gate that silently stops firing, a stochastic-tier KS test that drifts, an ADR amendment that turned out to be wrong. **Addition required:** ADR-0015 "Incident response and architectural rollback" with REQ-INC-0001 ("every CRIT incident produces an ADR amendment or a new ADR within N days"). **Consequence:** by PHASE-5 the team has no defined path from "we noticed X is broken" to "the architecture reflects what we learned."

ATTFUT-SER-013 — **Gate lifecycle (deprecate / archive) is absent.** ADR-0008 §1 inventories gates and §2 demands `gate-coverage`, but no rule covers gate removal. If ADR-0004 §4's anti-aliasing threshold proves wrong and the gate is retired, what happens to the REQ-CI-* covering it? Are deprecated gates kept in the inventory for historical audit? **Addition required:** REQ-CI lifecycle states (`active | deprecated | superseded`) and gate `gate-inventory-monotonic` (no silent deletes). **Consequence:** gate removal during PHASE-1 cleanup leaves orphaned REQ-CI-* in the matrix; coverage gate cannot distinguish "gate deprecated" from "gate never wired."

ATTFUT-SER-014 — **The amendment-authoring procedure is undefined despite being heavily relied upon.** ADR-0002 already has two amendments and ADR-0005 has one. The "Amendments are append-only" closing line in each is the entire procedure. There is no specification of: who triggers an amendment, what review it requires, how it relates to a CHG, whether the amendment row is the SoT or whether sections must be rewritten in place (both happen today). PHASE-5 explicitly says "any pain point... results in an ADR amendment, not a workaround" — making amendments a load-bearing mechanism. **Addition required:** `PROC-ADR-AMENDMENT` procedure plus gate `amendment-format-valid` (CHG link present, sections enumerated, summary atomic). **Consequence:** the most-used mechanism for evolving the architecture has the weakest discipline.

ATTFUT-PROC-015 — **`tools/trace/checkpoint_exemptions.yaml` is referenced but not spec'd.** ADR-0008 §6 names this file as the canonical list of gates exempt under `Checkpoint: true`. It is not in the PHASE-0 scope, has no REQ-CI-* covering it, and no gate verifies it stays in sync with the ADR-0008 §1 inventory. **Addition required:** include the file in PHASE-0 scope with an empty list, REQ-CI-* covering its existence, and a `checkpoint-exemptions-current` gate. **Consequence:** the only sanctioned bypass mechanism has no schema, no review trail, and no consistency check.

ATTFUT-PROC-016 — **`projects/<project>/openspec/` overlay semantics are undefined.** ADR-0001 §3 mentions "per-project specs extend root specs by reference; no duplication." How an overlay resolves, what conflicts mean, and what `tools/spec_lint` does with an overlay that shadows a root REQ are unspecified. PHASE-5 starts the first project — by then this must be specified. **Addition required:** ADR-0016 "Project-scoped openspec overlay resolution" plus REQ-SPEC-* covering overlay shadowing rules plus gate `overlay-no-shadow-conflict`. **Consequence:** PHASE-5 invents overlay semantics ad hoc and the precedent locks in before review.

ATTFUT-PROC-017 — **No retention/archival rule for the journal corpus.** ADR-0005 §5 makes the journal append-only and unbounded. By PHASE-5, a single long-running session can grow the journal to gigabytes. There is no rotation, compaction, or archival rule, and no signal at what size the journal is "too large" for `audit.py` to walk in bounded time. **Addition required:** REQ-AUDIT-0003 "journal rotation policy" plus an ADR-0005 amendment defining a `journal-rotation` gate. **Consequence:** by PHASE-5 the audit interface degrades silently from O(events) to operationally unusable.

ATTFUT-PROC-018 — **`tools/tea/` is named in three places but never spec'd.** ARCHITECTURE.md §3 lists it; ADR-0002 §7 says it is the sole authorised integration point for `_bmad/` references; PHASE-1 §5 names a "skeleton" with "integration with CI deferred to PHASE-2." There is no ADR scoping what `tools/tea/` is *for*, what surface it exposes, or what gates it owns. Yet it is the only sanctioned bridge between the BMAD substrate and project code. **Addition required:** ADR-0017 "TEA integration surface" with REQ-TEA-* requirements covering the exposed API plus gate `tea-api-stable`. **Consequence:** PHASE-1's "skeleton" becomes the de-facto specification, and PHASE-2's "deferred integration" lands without an ADR governing what it integrates.
